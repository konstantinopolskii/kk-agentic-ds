2. fab — floating action buttons (narrow viewports)

What it is
Three floating circle/pill buttons that appear only on tablet and phone. Left = nav (hamburger). Right = inspector (with comment count). Center = "add comment" pill, shown when text is selected.
Where
position: fixed, bottom corners. Hidden on desktop.
Selectors
.fab, .fab--nav, .fab--inspector, .fab--comment, .fab__count
Kill cost
You lose the ability to switch columns on tablet/phone. Without these, the narrow-viewport story breaks — there's no other way to swap views.

4. deck — fanned 3D card stack

What it is
A horizontal row of cards tilted away in perspective. The active one rotates flat and grows. Browse-then-pick.
Where
Used for theme presets, swatch flips, tier choices where the reader wants to skim before committing. Distinct from .card-stack (vertical, equal surfaces).
Selectors
.deck, .deck-card, .deck-card.active, .deck-card__select, .deck-card__check
Kill cost
High visual loss. The fan-and-rotate motion is one of the kit's signature affordances. No replacement pattern exists.
Visual decision — radius
Current .deck-card uses border-radius: 20px — a fifth, off-spec value. The kit allows 12 / 16 / 24 / 9999. The natural snap-to-grid replacement is 24 px. Side-by-side on the right: 20 (current) vs 24 (kit-spec). 24 looks slightly more pillowed; 20 is tighter. Both work.

Other defects on .deck-card (Cluster A): hard-coded #cccccc bg (line 1768) — fix to a kit token. 300 ms transition (line 1789) — snap to --dur-slow (320 ms) which is the closest kit token. 10 px check glyph (line 1830) — snap to 8 or 12. −190 px right-margin (line 1775) — this is the fan geometry; snapping it changes the visual rhythm. Keep or accept as a geometry exception?

12. .t-mono + .tag--inline — clean up and possibly unify

What's there now
.t-mono: kit-font (Commissioner) at weight 600, letter-spacing 0. Used for code, tokens, refs in body prose. The class name says "mono" but the font isn't monospace — it's a semantic marker for code-voice, not a font swap.
.tag--inline: surface chip + uppercase + bold + 0.8em + 0.2 px tracking. The 0.8em existed because uppercase reads visually larger; once uppercase goes, the size compensation goes with it.
Confirmed decisions
Keep font-weight: 600 on .t-mono as a documented exception to the 400/500/700 rule. Update tokens.json + components.md to name 600 as the kit's fourth weight, used only here.
Drop text-transform: uppercase from .tag--inline. Drop font-size: 0.8em back to body size. Drop letter-spacing: 0.2px (existed only to track the caps).
Naming question
"t-mono" lies — it's not monospace. "tag--inline" lies once uppercase is gone — it's not really a metadata tag anymore, it's a code chip. Both serve the same semantic: "this short phrase is code, not prose". The difference is just whether it sits on a surface (chip) or rides bare on the body weight bump.
Three rename options
A. Keep separate, rename t-mono. .t-mono → .t-code. .tag--inline stays as is (cleaned up). Two unrelated classes, honest names.
B. Unify under a code family. .t-mono → .t-code. .tag--inline → .t-code--chip. Modifier signals the relationship; both share the same root.
C. Single class, single modifier. Just .t-code for bare. Drop .tag--inline's surface treatment entirely — if a phrase needs more visual weight, that's a sign it should be a real button or callout, not an inline chip. Most aggressive simplification.

Recommendation: option B. Honest naming, the relationship is documented in the modifier, no behavior is lost.

Current

Bare body prose with a --space-4 token reference using .t-mono. Then a SHIPPING chip using .tag--inline in the wild.

Proposed — option B (rename + cleanup)

Bare body prose with a --space-4 token reference (now .t-code). Then an in progress chip (now .t-code--chip) — same surface, body size, no caps.

The chip variant feels right when the phrase needs to read as a thing, not as code. Bare .t-code is for actual tokens/refs; chip is for inline labels and statuses.

13. Animation inventory — every keyframe in the kit

Why this list
You said: blur is OK if (a) the admin authorizes it, or (b) it reuses an existing animation pattern. To make rule (b) enforceable, the existing patterns have to be named.
The blur question
One of the seven keyframes uses filter: blur(4px) as a transient effect (not a resting glass surface). That's the keyframe Cluster A flagged. With the new policy ("blur is OK if reusing existing pattern"), this keyframe becomes the canonical blur pattern — anyone wanting blur reuses it instead of inventing.
Total count
7 keyframes. All driven by the kit's motion tokens (--dur-fast 120ms, --dur-base 200ms, --dur-slow 320ms, --dur-long 760ms). Source: style.css, grep @keyframes.

Decision needed: codify these 7 as the kit's animation registry in canon/components.md § Motion? New animations require either reusing one of these or admin approval.

fake-caret-blink

1.06 s, steps(1, end), infinite. Pure opacity 1 ↔ 0.

Used on .field__fake-caret inside active comment-new and comment-thread cards. Marks where typing will land before the field has focus.

menu-in

--dur-fast (120 ms), --ease-out. Opacity 0 → 1 with scale 0.94 → 1.

Used on .comment__menu-popover when the kebab opens. Gentle pop from the top-right corner.

inspector-card-focus contains blur

--dur-long (760 ms), --ease-swing. From: scale(0.88) skewY(-3°) opacity(0.4) blur(4px). To: scale(1) skewY(0) opacity(1) blur(0).

Used on .inspector .card--interactive[data-state="active"] when a card promotes to active. The blur is transient — clears in 760 ms — not a resting glass surface. Under the new policy, this becomes the canonical "blur reveal" pattern. Anyone needing blur reuses this animation rather than inventing.

check-in

--dur-base (200 ms), --ease-spring. Opacity 0 → 1, scale 0 → 1.

Used on .deck-card__check when a deck card is chosen. The checkmark springs in once per choice — gestural confirmation.

reveal-from-left

Inline-attached by JS, 320 ms, ease-out. translateX(-12px) scale(0.96) → 0 / 1.

Used on the sidebar when it becomes visible (initial load or data-view="nav" swap).

reveal-from-right

Inline-attached by JS, 320 ms, ease-out. translateX(12px) scale(0.96) → 0 / 1.

Used on the inspector when it becomes visible.

reveal-from-below

Inline-attached by JS, 320 ms, ease-out. translateY(16px) scale(0.98) → 0 / 1.

Used on the doc/book column when it becomes visible.
