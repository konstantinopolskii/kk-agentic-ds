Comment persistence

Highlight a sentence below. A draft opens in the inspector. Type a comment, hit enter. Reload the page — the thread and the highlight come back.

useCommentStore ships a localStorage adapter by default. It writes a snapshot 200 ms after the last stack mutation; the next mount restores the thread stack and re-wraps doc highlights from each thread's own anchor metadata.

Three URL modes test the three adapter shapes: ?persist=on (default), ?persist=off, ?persist=memory. Current mode: on. Switch: on · off · memory

What persists
The full comment loop

Threads with anchor, author, and body. Replies inside a thread. Resolved threads (Approve from the kebab) and archived threads (Archive thread from the kebab). Drafts mid-typing. Highlights on the doc body, re-wrapped from the anchor metadata each thread carries on its dataset.

The snapshot is the stack's innerHTML plus a savedAt timestamp. No separate highlights array — data-kk-anchor-quote on the thread is the source of truth for re-wrapping.

Switching it off
For DB-backed apps

Apps that route the kk:comment event to their own backend should not also write to localStorage. Pass enabled: false to useCommentStore: 

useCommentStore(bookRef, inspectorRef, { enabled: false })
// or: { adapter: 'none' } — same effect, reads as intent at the call site.

Append ?persist=off to this page's URL (or use the switch above) to see it live. The stack stays empty across reloads. useCommentStore's returned clear() stays callable; it just has nothing to clear.

Custom adapter
Three methods, your store

Pass an object carrying load, save, and clear as the adapter option. useCommentStore calls load() once at mount, save(snapshot) on every batch of stack mutations (200 ms debounce), and clear() only when the consumer calls the composable's own returned clear(): 

const adapter: CommentStoreAdapter = {
  load()  { /* return a CommentSnapshot or null */ },
  save(snap) { /* snap: { v: 1, savedAt, stack } */ },
  clear() { /* remove */ },
}
useCommentStore(bookRef, inspectorRef, { adapter })

Append ?persist=memory to this page's URL (or use the switch above) to see a custom adapter at work. The snapshot lives in a plain JS closure. The stack survives in-tab navigation; a full reload resets it.

Public API
Three always-on methods

useCommentSecret's extractComments() walks the live stack and returns an array of threads with anchor metadata plus messages — same shape as the kk:comment event payload. copyComments() returns the same array and writes pretty-printed JSON to the clipboard. useCommentStore's clear() wipes the adapter and reloads.

Three buttons in the inspector run each method against this page. Open the browser console first; output lands there.

Anchor metadata
How highlights find their way home

Each thread mirrors the selection that made it. useCommentFlow writes data-kk-anchor-quote, -prefix, -suffix, and -section-slug onto the thread at draft commit. useCommentStore's restore reads those attributes, finds the section, walks text nodes for the first single-node match, and wraps it as <span class="highlight" data-comment-id="…">.

Selections that crossed element boundaries on first wrap (across a <strong>, for example) restore as a single-node match only. The thread restores intact; the doc-side highlight is partially lost. Same trade-off as the pre-2.0 kit.js implementation this composable replaces.
