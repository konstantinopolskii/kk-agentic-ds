Comment persistence

Highlight any sentence in this doc. A draft opens in the inspector. Write a comment and hit enter. Reload the page. The thread and the highlight survive.

The kit ships a localStorage adapter by default. Snapshots write 200 ms after the last mutation; the next load restores the stack and re-wraps doc highlights from each thread's existing anchor metadata.

Three URL modes test the three adapter shapes: ?persist=on (default), ?persist=off (kit stays cold), ?persist=memory (custom in-memory adapter, resets on full reload).

What persists
The full comment loop

Threads with anchor, author, and body. Replies inside a thread. Resolved threads (Approve from the kebab) and archived threads (Archive thread from the kebab). Drafts mid-typing. Highlights on the doc body, re-wrapped from the anchor metadata each thread carries on its dataset.

The snapshot is the stack's innerHTML plus a savedAt timestamp. No separate highlights array. The thread's data-kk-anchor-quote is the source of truth for re-wrapping.

Switching it off
For DB-backed apps

Apps that route the kk:comment event to their own backend should not also write to localStorage. Set enabled: false before kit.js loads: 

<script>
  window.KK = { config: { persist: { enabled: false } } };
</script>
<script src="../../js/kit.js"></script>

Append ?persist=off to this page's URL to see the off-switch live. The stack stays empty across reloads. KK.clearSavedComments() stays callable; it just does nothing.

Custom adapter
Three methods, your store

Pass an object carrying load, save, and clear. The kit calls load() once at init, save(snapshot) on every batch of stack mutations (200 ms debounce), and clear() only when the consumer calls KK.clearSavedComments(): 

window.KK = {
  config: {
    persist: {
      adapter: {
        load:  function () { /* return snapshot or null */ },
        save:  function (snap) { /* { v: 1, savedAt, stack } */ },
        clear: function () { /* remove */ }
      }
    }
  }
};

Append ?persist=memory to this page's URL to see a custom adapter at work. Snapshots live in a JS variable. The stack survives history navigation within the tab. A full reload resets the cache.

Public API
Three always-on methods

Three methods walk the live DOM each call:

KK.extractComments() returns an array of threads with anchor metadata plus messages. Same shape as the kk:comment event payload. KK.copyComments() returns the same array and writes pretty-printed JSON to the clipboard. KK.clearSavedComments() wipes the adapter and reloads.

Three buttons in the inspector run each method against this page. Open the browser console first; output lands there.

Anchor metadata
How highlights find their way home

Each thread mirrors the selection that made it. The kit writes data-kk-anchor-quote, -prefix, -suffix, and -section-slug onto the thread at draft commit. Restore reads those attributes, finds the section, walks text nodes for the first single-node match, and wraps it as <span class="highlight" data-comment-id="…">.

Selections that crossed element boundaries on first wrap (across a <strong>, for example) restore as a single-node match only. The thread restores intact; the doc-side highlight is partially lost. Same trade-off as the original inline-script implementation.
