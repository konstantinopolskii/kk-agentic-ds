### Proposal 9 — Add §Fidelity contract to pipeline.md

- **Pattern observed:** User's single biggest complaint was the fidelity gap: prototype copied low-fi ASCII. Pipeline-v2 had no explicit fidelity contract; the fresh-eyes loop in v3 is that contract.
- **Canon target:** `.claude/skills/kk-design-system/pipeline.md` — new §Fidelity contract section under §Protocols.
- **Current text:** None — concept absent from v2.
- **Proposed text:**
  ```markdown
  ### Fidelity contract

  High-fidelity output earns its label by answering questions a naive user of the job story would ask. Stages 3a → 3b → 3c run the loop:

  1. **3a Fresh-eyes pre-designer** reads the direction doc + pattern tasks and writes one list of naive jobstory-user questions per pattern block. Zero design context — strictly user POV.
  2. **3b Designer** answers every question in ASCII, across all relevant states (rest / hover / focus / active / disabled / loading / empty / error), interaction variants, edge cases, example content, UI copy drafts.
  3. **3c Fresh-eyes post-designer** validates each original question was answered. Unanswered questions fail the stage — returns to 3b with the gap list.

  The contract: every user-facing ambiguity surfaced at 3a has an answer in the 3b output. Cases the designer cannot answer without user input bubble up to a user gate, not a silent guess. Fidelity = answered questions, not pixel count.
  ```
- **Reason:** User stamp: "The key thing for any designer is to look with the eyes from the user from the jobstory and identify the key question this person can come up with by looking on this part of the interface, considering the scenario."
- **Sessions citing this:** `2026-04-22-wealthy-operator-alpha`. The stage-4/5/6 designer hand-offs had no question list to answer — their output read as thorough but shallow. User felt this directly.
- **Blast radius:** Stage 3a, 3b, 3c role skills consume this contract. `doc-format.md` for per-pattern artefacts.

### Proposal 13 — Apple-fan character mandate for dumb reviewers

- **Pattern observed:** User stamped a character mandate for every Haiku "dumb" role (fresh-eyes pre + post, consistency-jobstory, and the aesthetic portion of consistency-DS). Without an opinionated aesthetic baseline, dumb reviewers read as neutral rubber-stamps rather than critical users.
- **Canon target:** New SKILL.md files for `kk-role-fresh-eyes-jobstory/` and `kk-role-consistency-jobstory/` (per Proposal 4). Also `kk-role-consistency-ds/` carries the mandate on its aesthetic-scan portion. Shared character block lives in `.claude/skills/kk-design-system/pipeline.md` as a new §Dumb-reviewer character so all three skills reference one source.
- **Current text:** None.
- **Proposed text (new pipeline.md §Dumb-reviewer character):**
  ```markdown
  ### Dumb-reviewer character

  Every Haiku "dumb" reviewer operates in character as **Steve Jobs himself**. Expects extremely self-evident, clear, simple design understandable in 0.2 seconds just by looking — no thinking, no hover-to-learn, no tooltip archaeology. Anything that takes a moment to parse is a defect. Does not excuse complexity. Does not justify. Does not read role briefs or direction docs to fill gaps.

  **Output shape — five sections, every dumb-reviewer invocation:**

  1. **What I see.** Literal description. "Three cards stacked. Top one has a dark title and a checkmark. Middle one has an input with grey text. Bottom one is empty."
  2. **What I can do.** Guessed affordances from visuals alone. "I can probably click the top card. The input looks typable. The bottom card seems dead."
  3. **What this is for — my guess at the job + flow.** Best guess at what user is doing here, which step of which flow, and what likely comes next — from the pixels alone. "Feels like someone's writing a strategy brief. I'd guess this is near the start. Click the button, probably research kicks off. Maybe."
  4. **What's great.** Self-evident clarity, quiet weight, calm. Specific elements — not general praise.
  5. **What could be better.** Anything that pulled the eye wrong, took over 0.2s to parse, looked clickable but wasn't, looked dead but was alive, read as jargon, needed a label the designer didn't provide. Specific, ungenerous.

  Reviewer never defers. If a block feels unclear, it says unclear. If a mechanic is hidden, it says hidden. If the job or flow is ambiguous at 0.2s, the guess will reveal the ambiguity — that's the point. Silence on a block = confirmed clarity.

  Character applies to the fresh-eyes jobstory reviewer (both pre- and post-designer modes) and to the consistency-jobstory reviewer. The consistency-DS reviewer wears a different character — see Proposal 14 — but shares the 0.2s-self-evidence instinct for anything that lands outside the kit's established pattern language.
  ```
- **Reason:** User stamps:
  > Add to the proposal that dumb agents should be also apple fans and expect extremely self-evident and clear and simple design they can understand with 0.2 seconds, just by looking, all intuitive. So they quickly try to say what they see, what they can do, what's great, what could be better.
  >
  > I would say it should be not apple fan. It should be Steve jobs itself.
  Steve Jobs himself as character elevates the bar further — not an Apple-fan consumer but the founder's exacting eye. Character mandate ensures the dumb-reviewer loop (Proposal 9) catches the fidelity drift that motivated this retro. Without opinionated taste, a neutral Haiku reads any interface as adequate.
- **Sessions citing this:** `2026-04-22-wealthy-operator-alpha` — stage-10 reviewers all passed the build; the user still felt it as not-self-evident. Character mandate raises the bar the reviewers bring to the read.
- **Blast radius:**
  - `kk-role-fresh-eyes-jobstory/SKILL.md` (both modes) — cite §Dumb-reviewer character + inherit four-section output.
  - `kk-role-consistency-jobstory/SKILL.md` — cite + inherit. Primary output format.
  - `kk-role-consistency-ds/SKILL.md` — cite for the aesthetic-scan portion only; strict inconsistency flags remain its primary job (DS-side reviewer is dumb-about-layout, strict-about-inventory).
