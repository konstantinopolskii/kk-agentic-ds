# Retro — 2026-04-24 — markdown as source

## Observation

The initiative promised one home per fact. The session shipped three homes: `manifesto.md` at 685 lines, `index.html` as a runtime shell with hand-written pointer-card copy that itself contradicts the file it renders, and a `components.md` stub that points at the same manifesto from a third location. The renderer works. The architecture it renders contradicts the thesis it was built to prove.

The thesis was about duplication. Duplication was not removed; it was relocated and multiplied. The word "Components" appears twice in the sidebar — once under Inventory pointing at a pointer card, once under Matter pointing at the rendered manifesto §Components section. The word "Foundations" appears twice in the sidebar for the same reason. The pointer-card copy says "Nine in the kit" and "Nine components cover most SAAS flows" while the manifesto prose one scroll below says "Nine components plus pattern primitives." The drift the proposal opened by naming ("two radii" vs "four radii") was patched in manifesto prose and recreated in the signoff stat during Phase 3, where the engineer wrote fresh section counts that the next manifesto edit will break.

## Promises vs delivery

The proposal, in its own words (`proposals/2026-04-24-markdown-as-source.md` lines 20-28):

> One source per fact. Markdown as source. HTML as view. Prose lives in markdown only. HTML carries the shell, the six pointer cards, the inspector demo, and the renderer.

What shipped:

- Markdown as source: yes for the body of the manifesto. The 12 top-level sections render at runtime from `manifesto.md`. Renderer is real, smoke tests passed round 6, no console errors.
- HTML as view: partial. `index.html` is 359 lines, inside the 400-line ceiling the proposal named. The body renders from markdown. Good.
- Prose lives in markdown only: no. The six pointer cards in `index.html` carry hand-written prose that duplicates facts that also live in the manifesto. "Nine components cover most SAAS flows" (index.html line 114), "Nine in the kit" (line 109), "Eleven patterns from the first accepted prototype" (line 131), "Eight stages, eleven roles" (line 140). Each of these is a count claim that the proposal's §2 drift table treated as the canonical failure mode. The shell now owns four fresh duplicates.
- One home per fact: no. Three homes for component inventory: `manifesto.md § Components`, `components.md` stub, pointer card at `index.html` lines 106-121. The stub was chosen over deletion for one minor cycle — defensible — but the pointer card was not in the audit's duplication table and is the one that will drift first.
- Signoff stats: proposal said "Stats (abstract phrasing, no hardcoded counts per ruling 4)" (`phase-2a-audit.md` §6). Phase 3 shipped four hardcoded numeric stats: `12` sections, `9` components, `9` color tokens, `4` radii. The count the page claims ("12 top-level sections across Meanings, Matter, Protocols") is coupled to the current manifesto table of contents. Any next manifesto edit that adds or removes a top-level `##` silently invalidates the stat. The audit surfaced exactly this failure mode at row 11 ("Stale count — two drifts in one sentence") and the fix was to rebuild it.

The delivery proved the renderer. It did not execute the thesis.

## Iteration pattern — Phase 1's six Jobs rounds

Phase 1 was an infrastructure task: write a markdown-to-HTML renderer, smoke-test it, do not touch `index.html`. Estimated scope per proposal §Phase 1: one renderer file, one demo page, smoke matrix. The self-doc says the demo page went through six cold reads under the Steve Jobs character before passing.

The six rounds by defect class:

- **Round 1** (`phase-1-fresh-eyes-jobs.md`): 13 defects. Em-dashes in hero headlines. Rule 12 failures at every heading in the demo. Rule 14 on label-list pairs. Shell furniture competing with content.
- **Round 2**: 11 defects. Three hero-sized headings stack on the first viewport. Content labels ("Sample A/B/C") read as fixture names.
- **Round 3**: further rule 12 + rule 13 defects at the shell/article boundary.
- **Round 4**: same zone, tighter.
- **Round 5**: three defects. Hierarchy collapse between shell `.doc__part` (hero) and article `#` at t-display under offset=1. The fix bumped offset to 2 and dropped `.t-subtitle` from Bold to Medium to manufacture a weight step against `.t-caption--bold`.
- **Round 5.5** (phase-1h): that weight drop regressed every sidebar nav group title and every card sub-heading in the kit. Reverted t-subtitle to Bold.
- **Phase 1i**: the real fix — stop competing. Drop the shell `<h1>` and `.doc__part` from the smoke page, let each article render its own hero at offset=0. The problem was that the demo page carried hand-written shell chrome (page h1 + part label) around three rendered markdown articles; the rendered hierarchy collided with the shell hierarchy because both existed.
- **Round 6**: PASS. Zero defects.

Named honestly: the six rounds were not six discoveries. They were one discovery, postponed five times. Every round chased local fixes (offset bumps, weight swaps, padding-top rescues, `:has()` scope guards) against a structural problem the cold reader saw in round 1 and named again in round 5 defect 1: **the shell and the rendered content were fighting for the same hierarchy slot.** The fix was to delete the shell chrome. Everything before phase 1i was choreography.

The pattern this names: the pipeline's cold-read stages catch symptoms. The maintainer answers with local CSS patches. After enough rounds of patches, someone eventually deletes the thing that was causing the conflict, which is what round 1 told them to do.

Cost: six Jobs rounds × one audit + one rework each, plus a canon-weight change on `t-subtitle` that shipped and unshipped in the same phase, plus three `:has()` rescue rules in `style.css § Main document` that only exist to paper over the offset decision that was ultimately reversed.

The smoke test is now reliable. The route to that reliability burned five rounds re-discovering the structural answer.

## Architectural rot

Three kinds, ordered by which one bit first.

### Rules contradicting rules

1. **The signoff canon is unstable.** `manifesto.md § Components § Signoff` (line 316) now says: *"Two or four `.stat` children are valid shapes. Never three."* Phase 3 shipped four stats. `components.md §Signoff` (before retirement) previously said "Ship two stats, never three." The rule widened mid-initiative to accommodate what was already being shipped — Phase 2b resolved Q4/Q6 by relaxing canon to match the artifact. This is the pattern `manifesto.md § Protocols § Evolve` forbids: "either the rule updates or the code does — never both separately, never one silently." Here the rule relaxed to cover the code.

2. **The stat stays concrete where the proposal said abstract.** Ruling 4 in the proposal was clear: concrete counts are the default unless they are fragile. Phase 2a Q4 surfaced the signoff stats as the canonical example of fragile counts ("49 sections, 9 components, 9 color tokens, 2 radii" — "Stale count — two drifts in one sentence"). Phase 3 shipped fresh versions of the same fragile counts: "12 top-level sections across Meanings, Matter, Protocols" is the 2026-04-24 version of "49 sections across nine parts."

3. **The `index.html` pointer cards duplicate manifesto claims in the voice of the old index.** The six pointer cards carry `t-display` + `t-display--medium t-muted` subtitles that assert facts: "Nine in the kit," "Registry at the repo root," "Eight stages, eleven roles," "Values and swatches." These are the voice and structure of the pre-migration `index.html` doc sections that migrated to markdown. They are decorations on top of links. The cards would function as pure navigation with button labels alone ("Open fundamental," "Open the registry," "Read pipeline.md"). The prose above the button exists to restate what the page that the button opens will already say.

### Files duplicating files

- `components.md` stub (35 lines) points at `manifesto.md § Components`. Defensible for one minor cycle per the proposal; a real stub should be one line per target, no prose. The shipped stub has a heading per component, mirroring the manifesto's structure. If `manifesto.md § Components` adds a component, the stub gains a drift the next day.
- `index.html` carries a `#philosophy` pointer card idea in two places: pointer-card copy for §Components, §Patterns, §Pipeline, §Foundations, §Practice, §Distribution. The manifesto covers §Components, §Principles, §Foundations, §Protocols, §Pipeline in prose. The overlap set (Components, Foundations, Pipeline) renders twice on the same scroll: once as hand-written pointer card in the shell, once as rendered markdown section below. Two voices, same fact.
- The sidebar TOC contains duplicate entries for Components and Foundations because of this. Under "Inventory" → Components (pointer card, hand-written HTML). Under "Matter" → Components (rendered from markdown). Two anchors, one word, different surfaces.

### Mirrors

- Signoff stats mirror the manifesto TOC. The mirror is not declared. Every manifesto edit invalidates the stats silently.
- `index.html` pointer-card subtitles ("Nine in the kit") mirror the manifesto counts. Same problem.
- The sidebar nav groups "Inventory + Distribution" mirror the shell pointer cards. The "Meanings + Principles + Matter + Protocols + Surfaces" groups mirror the rendered markdown. Two halves of the same sidebar obey two different ordering principles. The user is expected to synthesize.

## "Really bad" — diagnosis

Reading the Phase 3 viewport at 1440×900 (`screenshots/phase-3-rewire-viewport.png`), first impressions:

- **Shell-vs-body fit.** The first screen shows `<h2 class="doc__part">Inventory</h2>` at 66 px bold, followed immediately by an `h2.t-display` "Components" inside a `doc__section` at 38 px, followed by a `t-display--medium t-muted` subtitle "Nine in the kit" at 38 px gray. Three heading ranks stack in the first viewport before any body prose. This is the exact round-1 Jobs defect from the smoke test ("three hero-sized headings stack on the first viewport with no hierarchy step") reintroduced at the page level. The smoke test fixed this by deleting its shell chrome. The index page kept its shell chrome.

- **Typography coherence.** `Nine in the kit` at 38 px Medium 500 muted. `Nine components cover most SAAS flows; every class is indexed in the patterns registry.` directly below at 16 px black Medium 500. Two phrasings of the same fact, in two sizes, in two colors, 40 px apart. The feedback rule "no muted by default" is honored for the body caption (black) but the pointer-card subtitle sits at muted gray — metadata? or the section's subtitle? The page does not decide.

- **Voice.** "Nine in the kit." "Registry at the repo root." "Eight stages, eleven roles." "Values and swatches." "Evolve here, inventory out." "The maintainer skill owns it." Six subtitles in six pointer cards, each trying to be pithy. They read as a second voice — decorative copy invented for the card pattern, separate from the manifesto's voice. The manifesto's voice says "A design system for AI-assisted product work. Four layers: meanings, perception, matter, pipeline." Factual, signed, first-person-plural. The pointer-card voice says "Values and swatches." Four-word tagline. A different writer.

- **Coherence of the page as one document.** The page claims to be a manifesto. A reader scrolls. They hit six pointer cards first, each naming a different artifact (fundamental, patterns.html, pipeline.md, tokens.json, maintainer skill). Then they hit a `# Manifesto` rendered hero and 12 `##` sections. Then a signoff. The pointer cards are not the manifesto. The rendered markdown is the manifesto. The shell is the manifesto's index page, but the index does not behave as an index — it behaves as a landing page with a manifesto attached below the fold.

- **The signoff stats.** "12 top-level sections across Meanings, Matter, Protocols." This is meta-prose — the page describing its own structure. The same reading fails the manifesto's voice rule ("if a paragraph doesn't end in a decision, cut it") and the AI-tells rule on mechanical self-reference. A signed document does not describe its own sections; it ends and signs.

"Really bad" reads to the eye as: the shell has its own opinions. It writes its own copy. It competes with the rendered content for voice and for hierarchy. The renderer did its job; the shell around the renderer did not keep up.

## Proposed changes

Specific. Edits to canon — the user accepts or rejects each; the maintainer bakes.

### Proposal 1 — Delete pointer-card subtitles

**Canon target:** `index.html` lines 106-203, six `<article class="doc__section" id="inv-*">` blocks.

**Current text:** Each pointer card carries `<h2 class="t-display">Card name<br /><span class="t-display--medium t-muted">Subtitle copy</span></h2>` + `<h3 class="t-title">` card heading + `<p class="t-caption">` body + button + `<p class="t-micro t-muted">` footnote.

**Proposed text:** Drop the outer `<h2 class="t-display">` with its `t-display--medium` subtitle. Keep the card heading (h3), the body caption, the button, the "Also:" footnote. Six subtitles delete: "Nine in the kit," "Registry at the repo root," "Eight stages, eleven roles," "Values and swatches," "Evolve here, inventory out," "The maintainer skill owns it."

**Reason:** Each subtitle duplicates a claim carried in the manifesto below. The cards are navigation. Buttons with labels are navigation. Subtitles are decoration that drifts.

**Blast radius:** CSS for `.doc__section > .t-display + ...` still applies to manifesto-rendered h2s. Sidebar nav group "Inventory" then points at six article ids — those stay. Visual: the page hero on first scroll becomes "Inventory" at 66 px followed directly by the "Components" card heading at 22 px. One step, not three.

### Proposal 2 — Rewrite the signoff stats to be structural, not count-based

**Canon target:** `index.html` lines 215-225, `.doc__signoff-stats` block.

**Current text:**
```
12 top-level sections across Meanings, Matter, Protocols.
9 components in the kit, enough to cover most SAAS flows.
9 color tokens, and not one more. No brand, no status color.
4 radii: 12, 16, 24, and 9999 for pills.
```

**Proposed text:** Two stats, not four. Content: the kit version and the sign date. `"Signed, kit v1.3.0."` and `"24 April '26, Tbilisi."` No counts. Counts belong in canon (manifesto + vars.css + tokens.json) where the single source is enforceable; they do not belong in a signoff the human hand-edits every session.

**Reason:** The signoff's purpose is "this document is done, here is who signed it, on what day, shipping which version." Count claims re-introduce the drift the initiative set out to close.

**Blast radius:** `manifesto.md § Components § Signoff` rule stays "two or four" — kit allows both. The index uses two. Other docs (strategy docs, call notes) may continue using four when the counts are genuinely stable for that artifact (revisions, edits pending).

### Proposal 3 — Retire `components.md` stub now, not "one minor cycle later"

**Canon target:** `skills/kk-design-system/components.md` + every SKILL.md canon-load list + non-skill references.

**Current state:** 35-line stub with 14 forwarding pointers, shipped with v1.3.0. Proposal §7 Q7 deferred removal to a 2.0.0 follow-up.

**Proposed text:** Delete the file. Update the 12 SKILL.md canon-load lists + 4 non-skill references to read `manifesto.md § Components` directly. The update was already done in Phase 2b (every consumer now has the new pointer); the stub only exists as a fallback for cached agent state. That cached state is not a real consumer.

**Reason:** The stub is the third home for component inventory. Keeping it guarantees drift. The whole initiative's premise was "one home per fact"; the proposal then reasoned itself into three homes. Remove the third.

**Blast radius:** Version bump from v1.3.0 to v2.0.0. The proposal §7 Q7 correctly named this as the only clean exit. 2.0.0 is not expensive — the consumers already read the right file.

### Proposal 4 — Collapse sidebar duplicate entries

**Canon target:** `index.html` lines 20-87, sidebar nav groups.

**Current text:** Seven nav groups. "Inventory" lists Components, Patterns, Pipeline, Foundations — pointers to shell cards. "Matter" lists Foundations, Components, Runtime — pointers to manifesto-rendered sections. Components appears twice. Foundations appears twice.

**Proposed text:** Drop the "Inventory" + "Distribution" groups as nav groups. The six shell cards are reachable by scrolling to the top of the page; the sidebar does not need to name them individually. The sidebar carries one structure: the manifesto's TOC.

Alternative: if the shell cards remain as a first-screen artifact, merge them under one group called "Open elsewhere" with six items (one per card), and keep the "Meanings / Principles / Matter / Protocols / Surfaces" groups pointing at the manifesto. Then no word appears twice. The duplication rule holds.

**Reason:** Duplicate nav items on a sidebar that is meant to read as the map of the page is a sign the page has two maps.

**Blast radius:** `.sidebar` CSS and `kit.js` scroll-spy unaffected — both observe `.doc__section` nodes and `[href="#id"]` anchors. This is a content edit only.

### Proposal 5 — Acknowledge the structural fight in the kit-shell-vs-renderer boundary as canon

**Canon target:** `manifesto.md § Foundations § Typography rhythm`, new rule appended.

**Current text:** Rules 1-14 plus rank-binding, title-leading, quote, caption-below-HTML addenda.

**Proposed text:** Add: *"Shell chrome does not compete with rendered content for heading rank. A page that carries markdown via `data-md-src` must either drop all shell-owned `h1`/`h2`/`doc__part` headings and let the markdown carry the full hierarchy (offset=0), or reserve higher ranks for the shell and set offset to match the depth already claimed. Two heading stacks on one scroll is the canonical hierarchy-collapse failure mode."*

**Reason:** Phase 1 rounds 1-5 rediscovered this rule. Phase 1i named it in prose. It should be canon so Phase N+1 does not rediscover it.

**Blast radius:** No code. Documentation rule. Future renderer demos + doc pages audit against it.

### Proposal 6 — Name the "shell duplicates body" failure mode in `§ Ship discipline`

**Canon target:** `manifesto.md § Ship discipline`, new bullet after the four existing verification bullets.

**Proposed text:** *"Before shipping a markdown-rendered page, grep the shell for every claim the rendered content also carries. A page that carries pointer-card copy, nav labels, or signoff stats that restate facts from the rendered body fails the one-home rule on ship. Remove shell-side claims or migrate them to the rendered source."*

**Reason:** This session's audit found 12 drifts. Phase 3 re-introduced 4 more. The failure mode is now proven — the cure is a ship-check.

**Blast radius:** Adds one verification step to the ship protocol. Cheap.

### Proposal 7 — Roll back Phase 3 before tagging

**Canon target:** the three commits on `main` that sit as v1.3.0-unreleased.

**Proposed action:** Do not tag v1.3.0 yet. The renderer infrastructure (Phase 1 + Phase 2b) ships clean. Phase 3's `index.html` rewrite is what introduced the pointer-card subtitle duplicates, the count-bearing signoff, and the double sidebar entries. Rework Phase 3 against Proposals 1-5 before the tag pushes.

Two ways to roll:
- **(a)** Tag v1.3.0 with Phase 1 + Phase 2b only — renderer on disk, `components.md` stubbed, manifesto expanded — and do not rewrite `index.html`. `index.html` continues as 767 hand-written lines until a second initiative rewrites the shell cleanly. Blast radius: the renderer does not yet render the canonical kit doc; only the smoke demo uses it.
- **(b)** Rework Phase 3 now. The shell changes are small (delete six subtitles, rewrite signoff, collapse two nav groups). Can land in one maintainer pass before the tag.

Recommendation: (b). The work is contained. The tag ships an honest v1.3.0.
