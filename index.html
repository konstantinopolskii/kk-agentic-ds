Philosophy
The essence

A product is a document. A document is a conversation. I want both to feel like one object — floating, quiet, responsive to attention. The rest of this system exists in service of that feeling.

Read it the way you’d read a good memo: once, carefully, then keep a copy next to your editor.

Material
Light on white

No gradients. No drop shadows. No glass. No blur. Depth comes from type and rhythm, not from imitating physical surfaces. If a card needs weight, give it more space. If a button needs authority, give it a bolder caption. We do not draw plastic on a screen.

Color
Nine tokens

Nine tokens, end of story. Two backgrounds, two surface tints, two hairlines, three text alphas. No brand blue. No status red. No accent teal.

If your copy isn’t unambiguous without color, rewrite the copy — not the palette.

Type
Inter, three weights

Inter — regular, medium, bold. Seven sizes from 64 px hero to 12 px micro on a tightened track. No italics outside of quotes.

Use weight and size for hierarchy. If a layout needs a fourth weight to read, the layout is wrong.

Space & rhythm
A 4px grid

Every gap, every padding, every margin snaps to a multiple of four. Most of the time: 4 8 12 16 20 24 32 48. Larger jumps mean larger semantic distance. If you need a 13 px gap, you need something else.

Radii
Two corners

Twelve pixels on the little things — buttons, tiers, keys. Twenty-four on the big things — cards, heading slots, anything that holds content. That’s it. A third radius forces the eye to ask “why is this one different?”. Usually it isn’t.

Motion
200ms ease-out

Motion is confirmation, not performance. Two hundred milliseconds, ease-out, on transform and opacity only. You should feel it, not watch it.

Two easing functions. Functional elements use ease-out — quick settle, no bounce. Playful elements use ease-spring — a brief overshoot before landing. If an element has a job to do, it doesn’t bounce. If it’s there to delight, let it spring.

Press feedback is universal: every tappable surface scales to 0.96 on :active. This is confirmation that the gesture landed.

If a user has prefers-reduced-motion on, we drop every duration to zero and let state change instantly.

Card
The surface that breathes

Every widget is a card. Cards come in two flavors. A static card is transparent at rest and fades to a 3% surface on hover or while an input inside has focus. Use it for info and light actions.

An interactive card has additional content hidden under a minimized version. It runs on a sticky active / minimized state. Active means gray with the collapsible revealed. Minimized means transparent with only the heading and a CTA. Triggering another interactive card in the same stack promotes it and demotes the previous one. One active per stack.

What qualifies a card to have a minimized state is hidden content, not inputs. A card that just fires an action has nothing worth keeping open — it stays static. But input focus still lights the surface on any card: if someone’s typing here, that card is active.

No borders. No shadows. Let hover, input focus, and state do the work.

Field
A line and a label

A label above, a value below, a 0.5 px hairline between this field and the next. On focus the hairline deepens from 10% to 20%. No boxes around inputs. The underline is the input.

Button
A caption on a surface

A bold 16 px label on a 12 px radius, full-width by default. Two variants.

The secondary sits on a 3% surface — this is the default. Use it for the CTA of a minimized card, a confirmation, any light action. Hover deepens. Press deepens further.

The primary inverts: black surface, white label. Use it for the executing action inside an active interactive card — where the user has made a choice and now commits. Its label is never the same as the minimized CTA: if the minimized card says “Pick tokens”, the primary says “Copy tokens”, not “Pick tokens” again.

Never outline-only. Never text-only. Never floating over content. One primary per card.

Tag
Metadata, not action

Two kinds. Metadata tags are bold secondary text — a date, a count, a status word. No pill, no surface, just the word in gray. Use them in rows with generous spacing between.

Inline tags appear inside running body text to call out a token or value: gray surface, normal weight, size inherited from surrounding copy.

Neither is ever clickable. If you feel tempted, you want a button.

Switch
A binary toggle

A track and a thumb. Off is the border tint; on is full text color. The thumb springs across with a brief overshoot — ease-spring at the fast duration. The track fills at the base duration.

Use it wherever a checkbox would sit: preferences, feature flags, opt-ins. Never use a switch if the result isn’t instant; anything requiring a save button should stay a checkbox.

Card stack
Browse by hovering or dragging

A horizontal deck where one card is always active and the rest tilt back in perspective. On desktop, hover promotes. On mobile, drag cycles. The active card springs to flat; the others collapse behind it.

One spring per transition, one active per stack — the same rule as the interactive card, extended to a browsable surface.

Navigation
A list with a hairline

A nav group is a bold subtitle, a list of captions, a 0.5 px hairline under the whole thing. The active item is in bold; nothing else marks it. No colored dot. No left bar. No pill. Just the weight change.

Comment
Name and body

A comment is a bold subtitle for the name and a caption for the body. Drafts use a Field component for the reply input — no new input shape. Comments live inside a comments card, never floating loose.

Signoff
Stats, signature, time

A few short stats in a three-column grid. If you only have four things to say, leave the third column empty — whitespace is the fifth stat.

One short paragraph with author, role, timestamp. A handwritten signature in SVG, right-aligned. This is how a document ends. Use it. Don’t invent a different ending.

Three columns
Nav, doc, inspector

Left column is the map. Middle column is the territory. Right column is the margin — actions, references, comments.

The middle column is the thing you read; the other two answer questions about it without ever becoming the thing.

Scrolling
Three independent scrolls

The shell is fixed. Each column scrolls by itself. This is what makes the app feel like one object instead of one long page: you can read without losing your place in the inspector, and work the inspector without losing your place in the doc.

States
One surface, three meanings

Three states, one visual: the 3% surface. Hover means “I’m pointing”. Input focus means “I’m typing”. Active means “this is the thing — there’s content revealed here worth staying with”. Button focus is not a card state — buttons fire and forget, so the card fades back as soon as the mouse leaves.

For interactive cards, active is sticky and exclusive: it stays until another card in the same stack is triggered — a click, a focus on an input, any action that semantically targets that card. Reading the doc, scrolling, tabbing through the sidebar: none of those touch inspector state.

When a card is triggered, the inspector glides so the newly-active card sits at the top — a 320 ms ease-out that starts immediately, not after anything else. The next thing you want is always the first thing you see. The replaced card collapses silently under the fold; you never see it close.

Don’t invent
Ask before you add

Every new component doubles the rules people have to remember. Before you reach for a new shape, ask whether a card with different contents will do. The answer is yes more often than you think.

If it’s not yes, bring the sketch to me.

The iPad feel
Split view, soft inside

Three panes in one frame, each with its own scroll. The left stays while you read. The right stays while you read. Only the middle moves. That’s the iPad split view.

Soft corners on every surface inside. Crisp edges outside where the document meets the screen. Not skeuomorphic buttons, not liquid glass. Rest and motion.

Narrow
Show where change happens

Three columns don’t fit on a phone. They almost fit on a tablet. One rule resolves it: show the column where the change is happening. Every other question about narrow layouts — what collapses, what swaps, what returns — follows from that sentence.

On phone, one column at a time. The doc is the default. Two circle buttons float at the bottom corners — a hamburger at the left to invoke navigation, a comment count at the right to invoke the inspector. When something changes in the inspector after a tap, the inspector takes over. Tap a nav item and the doc takes over again, because the scroll is the change you wanted. The buttons are always there; they don’t linger or disappear.

On tablet, two columns — the doc and the inspector. The sidebar is hidden by default; the hamburger button brings it in and it yields back to the inspector as soon as you’ve jumped. The doc never yields. The doc is the thing you read.

On desktop, three columns, as always. No caption bar. Nothing to swap.

No new components. No new states. The card’s active / minimized rules work exactly the same at any width; they run inside whichever column is currently mounted. A phone is a desktop with two columns collapsed.

Tone of voice
First person, signed

Every page is authored. Use “I” and “we”. Make claims. Use periods. If a paragraph doesn’t end in a decision, cut it.

If a page doesn’t deserve a signature, it doesn’t deserve to exist. Sign everything.
