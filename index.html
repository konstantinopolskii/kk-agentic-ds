Agentic Design System

A design system for AI-assisted product work. Four layers: meanings, perception, matter, pipeline. Output is screens with low cognitive load and a small component surface.

Agentic because AI does most of the drafting. The rules below exist so a junior agent (or a junior human) ships coherent work before human review.

49 sections across nine parts, from meanings to pipeline.
9 components in the kit, enough to cover most SAAS flows.
9 color tokens, and not one more. No brand, no status color.
2 radii everywhere. 12px for small things, 24px for cards.

Meanings

Philosophy
The essence

A product is a document. The document is the UI. The system treats both as one object. Every rule below serves that.

Why this exists
Business to pixels

Design verifies the business brief before pixels. If the brief has logical holes, the first task is to push it back. Every element on the final screen maps to a metric or a user problem.

Signal
What the user came here for. Must be obvious in under a second.
Noise
Everything else. Ruthlessly isolated, demoted, or removed.
Magic
Not placed. Emerges after signal is clear and noise is gone. Last step, never first.

Job stories
Context, motivation, value

A user hires a product for a job. One priority job per screen. Secondary scenarios are isolated or hidden until requested.

Formula
context + motivation = step → value
Example
Voice transcription during an interview. Context: the interviewer cannot look away. Motivation: don’t lose the question. Step: glance. Value: keep flow. So the screen is huge question cards that swipe themselves, every control hidden.
Anti-pattern
Designing from a component first, then inventing a job to justify it.

Time to value
Lead from aha to aha

Path length is measured in time to the first insight, not clicks. Cut steps where possible. Deliver value before the first tap when possible.

Principles

Pure signal
Modernism, not decoration

Utilitarian modernism, not postmodern complexity. Effort spent decoding the interface is proportional to how bad it is. Beauty is a side effect of clarity.

Expected patterns
Don’t make them think

Known patterns from the real world and common apps are already loaded in user memory. Reusing them costs nothing. Fighting them raises cognitive load for no gain.

Red button
Cancel or reset. Red is not used for anything else.
X in a corner
Closes the window. Not save, not hide, not minimize.
Back arrow
Goes back. Never opens a different section.
Hamburger
Navigation. Nothing else hides behind it.

Eighty / twenty
Fractal attention

Primary signal takes 80% of visual weight. Secondary fits in 20%. Fractal: applies to screen, panel, card, row. Empty space counts toward the 80%.

Screen level
In a Google Meet call, the participant video is 80%. The control bar is 20%.
Panel level
Inside that control bar, the central mic / camera buttons plus their empty space grab 80% of the panel’s attention.
Card level
Inside an active inspector card, the primary CTA plus the heading are 80%. Fields, metadata, secondary buttons fit in the remaining 20%.

Chunking
Seven, then split

Working memory handles about seven items. Beyond that, scanning degrades and the list becomes a wall. When a collection grows past seven, break it into categories. Each category becomes its own shorter list.

Fractal. Applies to bullet lists, numbered lists, spec rows, nav items, card stacks, menu items, component inventories. Any flat collection.

Sidebar
44 items
split into nine nav groups of 1–9 items.
Color tokens
9 tokens

Spacing tokens

Decision tree

Radical contrast
The enemy is gray mush

Gray mush is the default AI failure mode: muted everything, contrast nowhere, signal lost. The cure is hypertrophied contrast — hero 66px against body 22px, a 3× jump.

One distinction step between two elements. Bold vs regular, or big vs small, or black vs muted. Never all three at once. Never subtle.

The iPad feel
Split view, soft inside

Three panes in one frame, each with its own scroll. Only the middle column moves during reading. This is iPad split view, not a scrolling document.

Soft corners on every surface inside. Crisp edges outside, where the document meets the screen. No skeuomorphism, no liquid glass.

Foundations

Material
Light on white

No skeuomorphic surfaces. Depth comes from hierarchy and spacing, not from effects. Four things are forbidden:

Gradients
Solid tokens only. A gradient fakes light that isn’t there.
Drop shadows
A shadow implies physical depth. Depth comes from hierarchy, not pixels.
Glass
Translucency is a decorative effect with no informational value.
Blur
If the background matters, keep it legible. If it doesn’t, remove it.

Color
Nine tokens

Nine tokens: two backgrounds, two surface tints, two hairlines, three text alphas. No brand, no status colors, no accent.

Text is black. Muted and subtle are for metadata only — bylines, captions, hairlines. One distinction step per pair. Stacking color tiers creates gray mush.

If the text is ambiguous without color, rewrite the text, not the palette.

Motion
200ms ease-out

Default: 200ms × ease-out on transform and opacity only. Four easing curves, four roles.

Easings

--ease-out
Functional settle. Hover, focus, state flips. Default for everything.
--ease-spring
Playful overshoot. Switch thumbs, checkmarks, small confirmations.
--ease-swing
Penner’s easeOutExpo. Long exponential tail. Reveals and arrivals only.
--ease-in-out
Symmetric. For motion that continues past the frame.

Durations

--dur-fast
120ms
micro interactions, hover fades.
--dur-base
200ms
the canonical speed.
--dur-slow
320ms
inspector glide, promotion scrolls.
--dur-long
760ms
first-paint reveals, column arrivals.

Press feedback: every tappable surface scales to 0.96 on :active.

On prefers-reduced-motion, every duration drops to 0.01ms. State changes instantly.

Components

Card
The surface that breathes

Every widget is a card. Transparent at rest, a card fades to a 3% surface on hover, on input focus, or when promoted to active. No borders. No shadows. One visual, three meanings.

What qualifies a card to hold a minimized state is hidden content, not inputs. A card that just fires an action has nothing worth keeping open — it stays static.

Every direct child of a card sits on the same 24px rail. Raw paragraphs and lists get the rail without a card__body wrapper.

Three variants

Static
Transparent at rest. 3% on hover, focus, or input. For info and light actions.
Interactive
Sticky active/minimized state. Collapsible reveals on promote. One active per stack.
Shout
Inverted. Black surface, white content. One shout per column — if everything shouts, nothing does.

Static card

First in the stack — active, its surface showing.

Interactive card

Minimized underneath. Click to promote.

A shout inverts everything inside it — placeholders, dividers, even the primary button reverses polarity. Reach for it when a decision has to announce itself, not sit politely among the others.

Shout card

Inverted for the moments that matter.

Field
A row, a label, a value

A label and a bold value sharing one row. No box, no outline, no gap. A 0.5px divider sits between consecutive fields, never above the first or below the last.

Hover
The whole row fills with 3% surface. Nothing else moves.
Focus
Row inverts — black surface, white text. Caret flips to white.
Cursor
Pointer over the row, text over the input itself.
Commit
A ↵ glyph appears at the right edge once there’s content.

Button
A caption on a surface

A bold label on a 12px radius, full-width by default. Two variants, one rule: never outline-only, never text-only, never floating over content. One primary per card.

Secondary
3% surface. Default. Light actions, confirmations, minimized CTAs.
Primary
Black surface, white label. The executing action inside an active card.
Label pairing
Primary and minimized labels never match. “Pick tokens” → “Apply tokens”, not “Pick tokens” twice.

Tag
Metadata, not action

A tag labels a thing: date, count, status word. If it looks clickable, use a button instead.

Metadata tag
Bold gray text. No pill. For dates, counts, status words in rows.
Inline tag
3% surface chip, uppercase, tracked. For tokens or values embedded in text.

Switch
A binary toggle

A track and a thumb. Off is the border tint; on is full text color. Use it wherever a checkbox would sit — except when the result isn’t instant. Anything that needs a save button stays a checkbox.

Thumb motion
120ms × ease-spring. Brief overshoot, soft land.
Track fill
200ms × ease-out. The base speed.
Use for
Preferences, feature flags, opt-ins. Anything where change commits immediately.

3D card stack
Two easings, one gesture

A horizontal deck. One card active, the rest tilt back in perspective. On desktop, hover promotes. On mobile, horizontal drag cycles at a 30px threshold.

Browse without consequence — the commit button only appears on the active card. The rotation lands; the spread snaps.

Transform states

Resting
rotateY(52°) · rotateX(8°) · scale(0.75)
Active
rotateY(3°) · rotateX(0) · scale(0.95)
Perspective
120px, tight. Wider would look flat.
Origin
left center — every card fans from the same edge.
Rotation ease
300ms × ease-out. The tilt settles.
Margin ease
300ms × cubic-bezier(0.25, 1, 0.5, 1). The spread pops.

Comment
Name and body

Two shapes, one pattern. A draft card appears the moment the reader releases a selection in the doc. A thread card holds the conversation after a draft commits. Both live in the inspector stack; the same active/minimized rule that governs every interactive card governs them.

Tapping a mark in the body promotes the owning thread. On phone, the view swaps to the inspector so the thread lands in sight.

Every message’s kebab carries four actions. Reply and Delete are universal. Approve is gated on the thread’s last list message being agent-authored, and accepts that text as the replacement for the anchored quote; the thread collapses to a single checkmark row. Archive thread keeps the data, hides the card.

Lifecycle

Select
Range wraps in .highlight spans and stays selected, so ⌘/Ctrl+C still copies. A draft card pins at the top.
Draft
Shout card. Its highlights wear the inverted mark while open.
Commit
↵ replaces the draft with a thread card. Highlights return to 3%.
Minimized thread
First message + three-line ellipsis + last message.
Active thread
Shout. Every message plus a reply field.
Approve
Kebab item on any message. Available only when the thread’s last list message carries data-author-role="agent". Takes the agent text as the replacement for the anchored quote; thread collapses to a checkmark + snippet row.
Archive thread
Kebab item. Sets data-archived="true" on the thread; DOM retained, hidden from the stack. Distinct from Delete: data stays for future retrieval.
Discard
Escape. The draft, its spans, and the highlight all vanish.

Patterns

Three columns
Nav, doc, inspector

The left is the map. The middle is the territory. The right is the margin — actions, references, comments. The other two answer questions about the middle without ever becoming the thing itself.

Nav
200–260px, scales with viewport. Groups, items, the sliding bar.
Doc
1fr, centered, max 900px. The thing you read.
Inspector
340–420px, scales with viewport. Cards, threads, actions. The margin that talks back.

Scrolling
Three independent scrolls

The shell stays fixed; each column scrolls independently. Reading in the doc does not disturb inspector state. Working in the inspector does not disturb doc position.

States
One surface, three meanings

Three states, one visual: the 3% surface. Non-interactive surfaces do not mimic interactive ones.

Hover
Pointing. Fades back when the pointer leaves.
Input focus
Typing. Stays until focus moves.
Active
This is the thing. Sticky and exclusive — one per stack, until another is triggered.
Button focus
Not a card state. Buttons fire and forget.
Dismiss
Click empty space in the inspector. Whatever’s active collapses.

On card trigger, the inspector glides so the newly-active card sits at the top. Duration: 320ms × ease-out. Replaced cards collapse silently under the fold.

Forms
Compose cards that ask for input

Form cards follow one vertical rhythm: pick, scope, describe, commit. The primary action sits at the bottom. Fields and switches stack tight; sub-headings break the stack into groups with extra top space.

Flow
Pick a direction → scope the change → describe it → commit. Top to bottom, in that order.
Primary CTA
Last element in the card. Never in the middle, never above the inputs it commits.
Stacks
Fields and switches overlap internally — no gap between adjacent rows. Padding inside each row does the spacing work.
Sub-headings
Break a stack into groups. Automatic --space-4 top margin when they follow a field or switch.
Card captions
Roughly 90 characters. Two rows at card width. Describe the purpose; do not restate the title.
Field labels
Fit the 70px label column. Shorten the label; never widen the column.
One picker per decision
Don’t stack a deck and tiers that ask the same question twice. Choose the component that fits the cardinality, then stop.
Sub-headings that restate
Skip them. If the component below is obvious, the heading is noise.
Active vs minimized
Interactive form cards follow card rules: active inverts to shout in the inspector, minimized is transparent.

Narrow
Mobile-first polymorphism

Mobile-first. A phone screen is the base brick: one column, one job. Every widget stands alone at that width.

Desktop is not a separate layout. It is three mobile bricks side by side. Components do not change between widths; only the number of columns does.

On narrow viewports, show the column where change is happening.

Breakpoints

Phone
≤ 768px
One brick at a time. Doc is default. Two corner FABs: hamburger for nav, comment count for inspector.
Tablet
≤ 1024px
Two bricks: doc + inspector. Hamburger invokes nav; nav yields back on tap.
Desktop
≥ 1025px
Three bricks side by side. No FABs. Nothing to swap.

The swap has its own grammar. Outgoing columns vanish instantly. Incoming columns arrive widget by widget, each offset from the one above, from the direction the column lives on desktop — nav from the left, inspector from the right, doc from below. Motion with memory.

Reveal timing

Widget stagger
48ms between siblings, capped at 12 steps.
Column offset
160ms between columns. Doc leads, then nav, then inspector.
Duration
760ms × ease-swing. Fast start, exponential settle.
Scope
Only newly-appearing columns animate. What’s already on screen stays still.

This is shell behavior, not per-widget code. The orchestrator watches the data-view attribute and reveals every direct child of a newly-visible column in sequence. Add a card tomorrow — it animates in on its own.

Voice

Tone of voice
Collective, declarative

Short sentences, periods, no hedges. If a paragraph doesn’t end in a decision, cut it.

Voice
“We” where a subject is needed; no subject when the sentence stands alone. No “I”.
Shape
Short. Factual. Periods.
Bold
Headings only. Not body scanning.
Paragraph length
Over three sentences, move to a list or spec card.
Em-dashes
Forbidden in headlines. Rare in body.
Muted text
Forbidden by default. Allowed only for metadata (bylines, captions, hairlines) or when explicitly requested. Structural markers — list numerals, bullets, key cells — sit in full black.
Signature
Every doc ships signed.

Text as interface
Ninety percent of design is words

Screens are 90% text: labels, buttons, empty states, tooltips, errors. Form is secondary to meaning. If the label takes three reads, the design is wrong.

Labels and shape

Button labels
Imperative verbs. “Apply tokens”, not “Proceed”. Name the outcome.
Primary vs secondary
Labels never repeat. Secondary says “Pick tokens”; primary says “Apply tokens”.
Length
Cut every word that doesn’t change meaning.
Capitalization
Sentence case. No Title Case. No ALL CAPS.

Messages and hints

Errors
What went wrong + what to do next. No “Something went wrong”. No stack traces.
Empty states
One sentence of purpose + one action that populates.
Placeholders
Real examples, not labels. “sofia@kk.consulting”, not “Email”.
Tooltips
Metadata only. Features that need tooltips are UI bugs.

No AI tells
Unmask the agent

AI drafts leak the same tics. Cutting them is the fastest way to un-AI a passage. Based on the Wikipedia project’s inventory of signs of AI writing; every item below is forbidden in prose and UI text unless the user asks for it explicitly.

Words and verbs to cut

Filler adjectives
vibrant, pivotal, intricate, meticulous, robust, seamless, breathtaking, rich, comprehensive, holistic. Cut. If a noun needs an adjective, pick a specific one.
Buzzy nouns
tapestry, testament, landscape, journey, ecosystem, realm. Metaphors that say nothing. Cut.
−ing filler verbs
showcasing, fostering, highlighting, emphasizing, reflecting, contributing to, delving into. Drop the participle; name the action.
Copula avoidance
serves as, represents, stands as, features, boasts. Use is and has.
Transitions that pad
Additionally, Moreover, Furthermore, In conclusion. A period or a new paragraph already does the work.

Sentence shapes to cut

“Not just X, but Y”
Artificial urgency. Say both facts directly, one sentence each.
“Not A, but B”
Usually just say B.
Rule of three
Three-item adjective lists for fake completeness (“fast, reliable, and scalable”). Cut one, or all.
Elegant variation
Swapping synonyms to avoid repeating a word. Repeat the word if it’s the right one.
Em-dashes for punch
Forbidden in headlines. Rare in body. A period almost always does the same job.
Moralizing closers
“Ultimately, this reminds us…”, “In an ever-evolving world…”. Cut the closer; end on the last fact.

Structures and signals to cut

Summary blocks
“Key takeaways”, “In summary”, post-section recaps. The reader just read it. Cut.
Challenges and future prospects
Generic closing pattern. “Despite challenges, X continues to…”. Cut or replace with a specific fact.
Weasel attribution
“Industry reports show”, “experts argue”, “observers note”. Name the source or drop the claim.
“Such as” exhaustive lists
Naming five examples proves the category isn’t clear. Give one, or name the category.
Title Case headings
Sentence case in every heading. No Title Case Mid-Sentence Either.
Mechanical boldface
Bolding every instance of a term turns prose into an instruction manual. Bold the heading, not the concept.

Pipeline

Why a pipeline
Agentic, not art

AI agents draft most of our work. Without gates they produce gray mush, invent components, and ship the first plausible draft. Ten stages across three phases enforce breadth at the think phase, honesty at the hand-off phase, discipline at the build phase.

Nine role skills own the stages. Entry point matches scope — a typo enters at stage 9, a kit refactor enters at stage 1 plus 8, a new page walks all ten.

Think
Stages 1–3
Analyst, art-director, concept agents. Produces approved intent.
Hand-off
Stages 4–7
Three designer archetypes plus DS reviewer. Produces one human pick.
Build
Stages 8–10
Frontend engineer, UX copywriter, three parallel reviewers. Produces shipping prototype.
Meta retro
On demand
Reads session documentation, proposes canon updates. Never edits directly.

Think
Phase 1. Breadth before pixels

Three stages. The analyst decomposes the brief. The art-director opens the solution space. Concept agents produce low-fi options the human picks from.

Stage 1. Analyst
kk-role-analyst
Users, job stories, priority scenarios, open questions. Gates on human approval. No design until the brief holds.
Stage 2. Art-director
kk-role-art-director
Five or more directions, each with intent and guardrails. One direction is a fail. The human picks one direction; the rest archive.
Stage 3. Concept agents
kk-role-concept × 3–5
Parallel. Each produces ASCII flow mockups, a JSON component tree, and a shape-up framing. No code. The human picks one concept.

Hand-off
Phase 2. Three competing packages

Three designers run on the chosen concept in parallel. A DS reviewer writes a comparative analysis — not a ranking. The human picks the hand-off that goes to build.

Stage 4. Conservative
kk-role-designer-conservative
Strict kit inventory. No inventions, no reorganizations the kit's own patterns do not already contain. The safe floor the other two are measured against.
Stage 5. UX-driven
kk-role-designer-ux
Rearranges existing components to serve the job better. Layout frame stays kit-standard. No inventions.
Stage 6. Revolutionary
kk-role-designer-revolutionary
May break a rule with a matching diff. See revolutionary protocol.
Stage 7. DS reviewer
kk-role-ds-reviewer
Reads the three hand-offs. Writes a matrix plus one paragraph per hand-off: what each optimizes for, what it costs. Does not pick.

Accessibility floors are not negotiable inside the hand-off. Every package hits the 44 by 44 touch target, semantic HTML, contrast minimums.

Build
Phase 3. Structure, words, three reviewers

Build splits structure from voice so each gets its own pass. Three parallel reviewers gate the ship — code, words, system.

Stage 8. Frontend engineer
kk-role-frontend-engineer
Writes HTML, CSS, JS with kit classes and shared kit.js. Copy is placeholder comments — no real strings.
Stage 9. UX copywriter
kk-role-ux-copywriter
Fills every placeholder. Voice consistent across the prototype. Does not touch structure.
Stage 10. Frontend reviewer
kk-ds-frontend
Semantics, a11y, mobile, cross-browser, JS simplicity. Fails return to stage 8.
Stage 10. UX copy reviewer
kk-role-ux-copy-reviewer
Voice rules, AI-tells, button discipline, empty-state shapes. Fails return to stage 9.
Stage 10. Consistency reviewer
kk-ds-supervisor
Logic, 80 / 20, inventory. Fails return to the hand-off stage that owns the failure, not to the build.

Trigger
On demand only. No schedule, no automatic fire.

Accepted proposals route to kk-ds-maintainer. Rejected proposals archive in the proposal file with the rejection reason.

Documentation contract
Every stage self-documents

Each stage, as its final step, writes documentation/<session>/NN-role.md. A retro reads these files in order and reconstructs the entire decision arc without replaying the conversation.

Frontmatter
session, stage, role, input, output, gate. One line each.
Raw input
Verbatim user message where the stage received one. Not summarized, not restyled.
Artifacts
Pointers, not copies. Build files stay at their repo location; the doc links.
Rejected options
Archive in the session folder. Never delete. A retro often learns more from the rejected branch.
README
The analyst creates it at stage 1. Each subsequent stage appends its row to the file index.

Full template and frontmatter spec live in doc-format.md.

Revolutionary protocol
Break a rule only with a diff

The revolutionary designer at stage 6 may break a manifesto rule. Every broken rule ships with an entry in manifest-diff.md. Missing diff means the hand-off fails DS review automatically.

Diff entry shape

Canon source
File and section, e.g. components.md § What's forbidden.
Rule as written
Quoted verbatim from the canon file.
Proposed change
Specific replacement text ready to land in the canon file.
Reason
Tied to the job. Not aesthetics.
Blast radius
Other canon files and live prototypes the change touches.
Rollback
What reverting looks like.

Two human paths at stage 7. Reject the diff — the revolutionary hand-off falls back to the UX-driven variant. Accept the diff — kk-ds-maintainer runs before stage 8, updates the canon, bumps the kit, then the build proceeds.

Accessibility floors (44 by 44 targets, semantic HTML, contrast) are outside the protocol. The revolutionary cannot trade those away.

How to reuse
Pick, don’t invent

Start from existing components, not a blank canvas. The opening question is “which is closest?”; then adapt. A card with different contents usually solves it.

Structural needs

Need a container?
Card. Static if it just shows; interactive if it holds content under a minimized state; shout if the moment matters.
Need an input?
Field. Row variant for label + value; plain for free-text.
Need a commit action?
Button. Primary if it executes a decision, secondary otherwise.
Need a toggle?
Switch if change commits instantly. Checkbox if it needs a save.
Need to show metadata?
Tag. Metadata variant in rows, inline variant inside text.

Explanatory needs

Need a picker?
3D card stack. Browse without consequence, commit on the active card.
Need to explain a rule?
Spec table inside a card. Key on the left, value on the right.
Need to list steps?
Bulleted or numbered list. See Type.
Still nothing fits?
Ask. See How to add.

Don’t invent
Ask before you add

Every new component doubles the rules to remember. Before adding a new shape, check whether a card with different contents works. Usually it does.

If not, bring the sketch. See How to add for the process.

How to add
Small steps, in order

New components are expensive. Each one adds rules for everyone else to hold. When unavoidable, follow the process below.

The process

  1. Check the inventory. Ninety percent of the time, something already fits.
  2. Try the closest existing component. Stretch it until it tells you honestly that it won’t work.
  3. Write a one-page proposal: what it is, why the existing kit fails, where it lives, which tokens it uses.
  4. Get sign-off before writing CSS. This is the step that keeps the kit small.
  5. Add tokens first, pixels second. If you need a value that isn’t in vars.css, the value earns a token.
  6. Build the smallest version that works. No variants until the base is used.
  7. Add a live demo in this document under Components. If it can’t be shown here, it isn’t done.
  8. Sign the PR. The kit is a signed document; every change carries an author.

Adding a variant of an existing component follows the same process, minus the inventory check. Two variants of the same component are cheaper than a new component.

How to evolve a rule
When the manifesto drifts

Rules in this document are canonical. Code that disagrees with the manifesto is wrong unless the manifesto is. Either the rule updates or the code does — never both separately, never one silently.

On conflict

  1. State the conflict out loud. Name the rule, name the code. No silent overrides.
  2. Decide which side is right for the product today, not for the version that wrote the rule.
  3. Update both in the same PR. The rule moves in the doc; the code moves in the kit.
  4. Log the conflict in the Backlog section until it ships.
  5. Sign the change.

Unresolved issues live in Backlog openly. Hidden issues are worse than visible ones.

What to think about
Before you ship

The checklist a junior designer (or a non-designer) can run before pushing a change. Each box is a rule from elsewhere in this doc; the checklist is the quick reference.

Before pixels

Job story
Can you name the one job this screen does? If no, stop.
Eighty-twenty
Primary signal takes 80% of the attention. The rest fits in 20%. Check at every nesting.
Contrast
Hypertrophied, not polite. No gray mush. One distinction step between two elements, and make it count.
Expected patterns
Red = cancel. X = close. Don’t fight cognitive defaults.
Mobile first
Phone screen is the base brick. Desktop composes from it.

Tokens and grid

Grid
Every size, gap, padding on the 4px grid. No 13px gaps.
Tokens
Tokens before pixels. If you wrote a raw value, replace it with a token, or add one.
Color
Only the nine. No brand blue, no status red.
Radius
Only 12 or 24. Anything round is a pill.
Motion
200ms × ease-out by default. Transforms and opacity only.

Interaction and access

States
Hover, focus, active all land on the 3% surface. One visual, three meanings.
Reduced motion
Duration drops to 0.01ms. The feature still works.
Keyboard
Tab order is logical. Every focus state is visible.
Touch
Targets at least 44 × 44 px. Hover-only affordances have a focus equivalent.

Text and voice

Text
Labels are imperative. Errors say what to do next. Empty states lead to the action.
Voice
“We” or no subject. Periods. Cut anything that doesn’t end in a decision.
Signature
If it ships, it’s signed.

Distribution

Skill for agents
The kit inside the agent’s mind

The kit ships two ways in one package. Humans get CSS, fonts, behavioural JS, and this document. Agents get a Claude skill that constrains their output to the kit before the first character prints. One source of truth, one version number, two consumers.

A skill is not a reference. The agent does not read it and then freestyle. The skill is loaded into context on every invocation and tells the agent what it may emit, what it may not, and in what order to think. Invented components and off-grid values are rejected at supervision.

Skill layout

SKILL.md
entry
Trigger description, hard rules, pipeline protocol. Claude loads this automatically when a UI or document task matches.
tokens.json
machine-readable
Every legal color, space, radius, type size. The agent consults this without parsing CSS.
manifesto.md
principles
Philosophy, 80/20, chunking, contrast, foundations. The why behind the tokens.
components.md
inventory
Every kit component with its class contract and a copy-pasteable HTML snippet.
voice.md
tone
Shape rules, label rules, the full AI-tells inventory. Applied to every word the agent writes.
pipeline.md
protocol
The five-stage agentic flow documented in Pipeline, expanded for agent use.
patterns/
recipes
Canonical use-case layouts. strategy-doc.md covers the Wealthy pipeline’s output.

Skill as pipeline

Three skills ship in the package. The main one owns stages one through three. Stage four runs in two roles: the frontend rewrites, the supervisor audits. The human owns stage five.

kk-design-system
stages 1–3
Analyst, UX, UI. Invoked on any doc or UI task. Runs hypothesis defense, iteration, and contrast boost.
kk-ds-frontend
stage 4, rewrite
Runs on the stage-three draft. Rewrites semantics, accessibility, JS, mobile, and cross-browser code without moving a pixel.
kk-ds-supervisor
stage 4, audit
Runs on the rewritten draft. Audits logic, 80/20, and inventory. Returns failures to the stage that owns them.
Human
stage 5
Magic injection and signoff. The agent stops; the author lands.

Claude plugin
marketplace
.claude-plugin/plugin.json lets consumers install via /plugin install without npm.
Manual copy

Install and update
One package, two consumers

The kit ships as @kk/design-system. One package carries tokens, components, fonts, behavioural JS (js/kit.js), and the Claude skills. Humans import the CSS and include the script; agents load the skill. Both share one version.

Consumers pin a version; they never edit the kit in place. Local edits in node_modules or a submodule get wiped on the next install. All changes return to this repo as PRs.

Install in a project

npm
primary
npm install @kk/design-system. CSS imports from the package; postinstall symlinks skills/ into .claude/skills/. One command, both consumers wired.
Claude plugin
skill-only
/plugin install kk-design-system. For projects that want the agent rules without the runtime CSS (docs, chat adapters).
Git submodule
escape hatch
Last resort. Works when npm is not an option. Painful at month three — use only if the other two fail.

Versioning

Scheme
Semver. Major: breaking token or component removal, trigger-phrase change in SKILL.md. Minor: additive components, tokens, skill recipes. Patch: fixes, copy edits.
Pin
Lockfile (package-lock.json) or version field in plugin manifest. Never float.
Changelog
Every release names what was added, what was removed, what moved. Lives at CHANGELOG.md and in the GitHub release body.
Trigger phrases
Changes to a skill’s description are breaking even if no classes move — the agent may stop invoking the skill. Treat as major.

Update flow
Tag, publish, PR, merge

Updates travel in one direction: this repo, then out. Consumer repos do not edit the kit; they receive it. A tagged release here triggers a PR in every consumer repo that depends on the package.

Release in this repo

  1. Edit tokens, components, skill files. Commit to a branch.
  2. Update CHANGELOG.md with the diff: added, removed, moved.
  3. Bump package.json and the version in .claude-plugin/plugin.json in lockstep.
  4. Merge. Tag vX.Y.Z on main.
  5. GitHub Actions runs .github/workflows/release.yml: publishes to the registry, creates a GitHub release with the changelog, dispatches an event to every consumer.

Consumer picks up the release

Renovate / Dependabot
automated
Watches @kk/design-system. Opens a version-bump PR in the consumer with the changelog pasted into the body.
repository_dispatch
automated
Release workflow fires a kk-design-system-updated event. The consumer repo listens, runs a Claude agent with the changelog as input, and drafts call-site edits before opening the PR.
Human review
mandatory
No auto-merge. The author reads the diff, runs supervision, ships.

Why not auto-merge

A skill update can change trigger phrasing, component names, or voice rules. The agent will adapt the consumer code, but judgment — is the new label right for this product? — belongs to the maintainer. Bots open the PR. Humans merge it.

Never hand-edit the kit inside node_modules or a submodule. The next install wipes it. Land changes here, tag, release.

Guards and checks
Who owns what, who verifies

Agents draft fast and forget rules. Three guards keep the system honest: a file boundary that locks the kit, a linter that catches forbidden values, and the pipeline’s supervisor agent that catches semantic drift. The human signs off at the end.

Ownership

Kit files
vars.css, style.css, js/kit.js, fonts/. Maintainer only. PRs against the kit repo, not the consumer project.
Project code
Consumer. Free to compose kit components; never modifies the kit itself.
Content
Author. The words inside any instance of the document.

Who checks what

Linter
automated
Off-grid spacing, off-palette colors, forbidden radii, light weights, muted text outside metadata.
Supervisor agent
automated
Job-story fit, 80 / 20 at every nesting, invented components, drift from the component index.
CI gate
automated
Blocks PRs that fail lint or supervisor. No merges past a red check.
Art director
human
Judgment calls, tone, magic injection. Last signoff before ship.
Maintainer
human
Guards the kit itself. Approves rule changes in the manifesto before code lands.

Agents never commit to the kit repo directly. They draft against a consumer project; the supervisor agent sends every failure back to iteration. Only the pipeline’s final step pushes work to a human.

Auto-sync inspector comments to Claude

Goal
Let Konstantin leave comments in the inspector; have Claude pick them up across sessions and improve the doc automatically.
Option A
File System Access API. Browser writes threads to comments.json. No server, but Chromium-only and requires a local HTTP server (not file://). First commit needs a file-picker click.
Option B
Local Node server + SQLite + MCP. Browser posts each comment; Claude reads and resolves through an MCP tool. Works everywhere; needs a running process.
Option C
Manual export button that downloads a file. Simple. Extra click each time.
Round-trip problem
When Claude marks a thread resolved, the browser doesn’t know. Next write overwrites the resolution unless the browser reads the file first, hides resolved threads on load, and merges state on write. Doable, not cheap.
Parked
Current volume of review feedback doesn’t justify the plumbing. Writing notes directly into the doc or conversation works for now.
Revisit if
Reviews start coming in across many sessions and manual capture gets lossy. Then build Option B — server + SQLite + MCP — as the path that covers all browsers and handles the round-trip cleanly.
